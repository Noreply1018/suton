# Suton / 溯通

Suton / 溯通 是一个面向考试复习与课程速通场景的资料溯源工具。它不试图替代学生学习，也不把自己包装成无所不知的 AI 老师，而是专注解决一个具体、高频且耗时的问题：当学生直接从做题开始复习时，如何快速找到这道题对应课本、PPT、讲义或老师资料中的哪一部分内容。

Suton 的核心价值不是制造更多泛化解释，而是降低“题目到资料”的检索成本，让刷题、查书、补知识这三个动作连成一个顺畅闭环。

## 产品定位

Suton 的核心思路是“从题目出发，反向打通资料依据”。

v0.1.0 中，用户上传带文字层的课程资料 PDF，并手动输入或粘贴一道题。系统通过 PyMuPDF 文字层解析、Redis + RQ 后台处理、pgvector 语义索引，把资料转化为可检索的最小知识库，并返回可追溯的资料依据。

当用户面对一道不会、不熟或想确认依据的题目时，Suton 不会优先给出脱离资料的泛化讲解，而是帮助用户快速定位这道题在指定资料中的相关文件、页码和原文片段。学生可以立刻回到考试依据本身，看到“这道题到底从哪里来、应该看哪几页、对应哪部分内容”。

## 解决的问题

- 资料很多：教材、PPT、讲义、实验指导书、题库和历年题分散在不同文件中。
- 考试范围明确：复习必须以课本、老师材料和课程资料为准。
- 时间有限：人工翻找题目对应知识点、页码和例题成本很高。
- 做题先行：学生常常从题目开始复习，但题目与资料依据之间缺少直接连接。
- 依据优先：相比泛泛解释，很多场景更需要知道“原文在哪里、考试依据是什么”。

## 工作流

1. 上传课程资料
   用户上传带文字层的教材、课件或讲义 PDF。

2. 构建课程知识库
   系统使用 PyMuPDF 提取文字层，按页或页内段落切块，并建立 pgvector 语义索引。

3. 输入题目
   用户手动输入或粘贴一道题，系统保存题目文本。

4. 从题目反查资料
   系统在指定资料范围内检索相关文件、页码和原文片段。

5. 回到依据复习
   用户根据定位结果回到课本、PPT 或讲义原文，补齐知识点并继续刷题。

## 核心能力

- 文档解析：处理带文字层的教材、课件和讲义 PDF。
- 语义索引：将资料文字层内容转化为可按题目语义检索的知识库。
- 手动题目：保存用户手动输入或粘贴的一道题。
- 资料溯源：从题目反向定位到相关文件、页码和原文片段。
- 范围约束：优先在用户指定的课程资料和考试范围内查找依据。
- 来源约束：没有文件、页码和原文片段的结果不得展示。

## 适用场景

- 考前复习：快速确认题目对应的教材章节、PPT 页码或讲义段落。
- 课程突击：在时间有限时，从高频题目反向补齐关键知识位置。
- 开卷资料整理：把分散资料与题目建立连接，减少临场翻找成本。
- 题库反查：将练习题、历年题与课程资料中的公式、概念和例题关联起来。
- 资料密集课程：适合教材、PPT、讲义、实验指导和老师补充材料较多的课程。

## 产品边界

Suton 不是一个替你学习的工具，而是一个帮你更快找到学习依据的工具。

它不会把目标放在替代学生理解、记忆和练习上，也不会默认输出脱离课程资料的“万能答案”。Suton 更关注依据、页码、段落、公式、图表和例题之间的可追溯关系，让学生能够回到原始资料中学习，并基于明确来源完成复习。

## 命名含义

Suton 有“速通”的发音意味，强调快速抵达目标知识位置。

中文名“溯通”表达了“溯源”和“打通”的双重含义：从一道题向上追溯到资料来源，再把分散的课本、讲义、PPT 与试题之间的关系打通。

## 第一版方向

第一版建议优先做 Web App，暂时不做 APK。

原因是 Suton v0.1.0 的核心不是手机拍题，而是上传资料 PDF、手动输入题目、建立索引，并在做题时快速定位资料依据。这类工作天然依赖 PDF 页码跳转、左右分栏、资料上传和后台索引构建，电脑端 Web 体验更顺手，工程上也更容易把核心链路打穿。

APK 属于后续补充形态，用于拍照上传题目、移动端碎片化刷题或考前查看资料位置，不属于 v0.1.0。

v0.1.0 的一句话定义是：

> 上传带文字层的课程资料 PDF，手动输入一道题后，系统返回资料中的相关文件、页码、片段和原始 PDF 入口。

v0.1.0 页面只回答一个问题：

> 这道题应该去资料哪里看？

## MVP 范围

v0.1.0 保持克制，优先验证“题目能否准确定位到资料依据”。

v0.1.0 只做：

- 上传资料 PDF。
- 使用 PyMuPDF 解析 PDF 文字层。
- 按页保存提取文本。
- 默认按页切块，单页超过 2000 个 Unicode 字符时在页内按段落切分。
- 为资料块建立 embedding 和检索索引。
- 用户手动输入或粘贴一道题。
- 返回最相关的最多 5 个资料位置，包括文件名、页码、命中片段和原始 PDF 入口。

v0.1.0 不做：

- APK。
- AI 讲解。
- 错题本。
- 知识图谱。
- 学习计划。
- 自动总结。
- 社区题库。
- 完整自动切题。

v0.1.0 不做试题上传：只支持上传资料 PDF、建立索引、手动输入题目并返回资料页码、片段和原始 PDF 入口。只要这个链路成立，产品核心价值就成立。

## v0.1.0 页面

固定保留三个核心页面。

### 项目页

一个考试或课程对应一个项目，例如“高性能计算速通”“电磁场数值分析速通”“通信原理期末复习”。

项目下管理资料文件和手动题目：

- 资料文件。
- 手动题目。

### 资料上传页

用于上传带文字层的课本 PDF、课堂 PPT 导出的 PDF、老师讲义 PDF 和实验指导书 PDF。

上传后展示处理状态：

```text
文件              页数  处理状态  索引状态
高性能计算课件.pdf   82  完成      完成
实验指导书.pdf       31  完成      完成
```

### 题目输入页

用于手动输入或粘贴一道题。v0.1.0 不上传试题 PDF、作业题 PDF、历年真题或图片题。

### 做题与查依据页

这是主页面。固定采用左右分栏：

- 左侧展示题目。
- 右侧展示资料依据。

右侧结果不写长篇解释，只展示可追溯依据：

```text
1. 《实验指导书》第 7 页
   命中段落：关于二维块循环分布和进程网格的说明
   操作：打开 PDF 第 7 页

2. 《课堂 PPT》第 15 页
   命中内容：MB、NB 对通信与计算负载的影响
   操作：跳转 PDF 第 15 页
```

页面上最多保留一句提示：

> 系统判断本题主要关联这些资料位置，请优先查看。

## 技术方案

v0.1.0 固定采用 Web + 后端处理服务 + 数据库 + 文件存储的架构。

```text
前端：Next.js + React + Tailwind CSS + pnpm
后端：Python FastAPI
数据库：PostgreSQL + pgvector
后台任务：Redis + RQ
文件存储：本地 uploads
PDF 解析：PyMuPDF
OCR：v0.1.0 不实现
LLM：v0.1.0 不实现
Embedding：DashScope text-embedding-v4，1024 维，OpenAI 兼容接口，凭据为 DASHSCOPE_API_KEY
Reranker：v0.1.0 不实现
```

技术取舍原则：

- 前端优先服务上传、预览、左右分栏、题目卡片、资料依据展示和页码跳转。
- 后端负责文件接收、PDF 文字层解析、题目处理、资料切块、embedding、检索和返回依据。
- PostgreSQL 管项目、文件、页码、题目、chunk 和关联结果，pgvector 负责第一版向量检索。
- v0.1.0 不引入 Qdrant、Elasticsearch 或其他独立检索服务。
- PDF 解析和索引构建必须通过 Redis + RQ 后台任务处理，不阻塞用户页面。

## 文档解析与 OCR 策略

v0.1.0 不实现 OCR。第一版只处理带文字层的 PDF：

- PDF 有文字层时，使用 PyMuPDF 直接提取文本。
- 扫描版 PDF、图片型页面、拍照讲义和截图题目，在 v0.1.0 中标记为不支持。
- 不引入 PaddleOCR、Tesseract、Docling 或 pdfplumber。

v0.1.0 不追求公式、表格和复杂排版结构理解。只要系统能基于文字层把题目定位到正确页码和大致段落，用户就可以回到原始 PDF 自行查看细节。

## 核心处理流程

### 资料处理 Pipeline

1. 保存原始资料文件。
2. 保存原始 PDF，并在结果中提供可回到原始 PDF 的入口。
3. 判断页面是否有文字层，有则直接提取，没有则标记为不支持。
4. 保存每页文本和页码。
5. 默认按页生成 chunk；单页文本超过 2000 个 Unicode 字符时，在该页内按段落切分并合并为不超过 2000 个 Unicode 字符的 chunk。
6. 为每个 chunk 生成 embedding。
7. 将 chunk、页码、原文片段和 embedding 入库。

关键数据对象：

```text
documents
- id
- project_id
- filename
- type
- status

document_pages
- id
- document_id
- page_no
- raw_text

chunks
- id
- document_id
- page_id
- page_no
- text
- section_title
- embedding
```

### 试题处理 Pipeline

v0.1.0 不处理试题 PDF 上传、自动切题、题目截图或题号识别。题目只来自用户手动输入或粘贴。

固定验证题目保存在 `tests/fixtures/question.txt`，人工摘录自 `tests/fixtures/question-source.pdf`。试题来源 PDF 只作为人工证据，不进入处理链路。

关键数据对象：

```text
questions
- id
- project_id
- text
- created_at
```

## 检索与重排

v0.1.0 固定使用 pgvector 进行 embedding 相似度检索。关键词检索不参与结果排序，不得替代 embedding 检索。

embedding 固定使用阿里云百炼 DashScope `text-embedding-v4`，输出 1024 维 dense float 向量。后端通过 OpenAI Python SDK 调用 DashScope OpenAI 兼容接口 `https://dashscope.aliyuncs.com/compatible-mode/v1` 的 `/embeddings`，凭据从 `DASHSCOPE_API_KEY` 读取。

检索链路固定为：

1. 读取用户输入题目文本。
2. 使用已固定的 embedding provider 为题目生成向量。
3. 使用 pgvector 在 `chunks.embedding` 上按相似度召回候选 chunk。
4. 过滤缺少文件名、页码、片段或 chunk 来源的候选结果。
5. 按 pgvector 相似度分数返回前 5 个来源明确的资料位置。

每条结果必须绑定来源：

```text
question_matches
- id
- question_id
- chunk_id
- score
- reason
- rank
```

前端展示结果时必须包含：

- 文件名。
- 页码。
- 原文片段。
- 原始 PDF 页码入口。
- 命中原因；v0.1.0 中命中原因必须由确定性规则生成，例如“pgvector 相似度命中”，不得由 LLM 生成。

如果一条结果没有明确资料来源，就不应展示。

## 大模型使用边界

Suton 不是 AI 答题器，大模型不应站在台前直接讲题。

v0.1.0 不使用大模型做切题、关键词提取、候选重排、相关性判断、讲解生成或命中理由生成。

没有 `document_id`、`page_no`、`chunk_id`、`source_text` 和 `score` 的结果，不进入前端依据列表。

## v0.1.0 推进顺序

v0.1.0 只按以下顺序推进：

1. 固定工程骨架、Make 验证命令和本地环境核验。
2. 支持项目创建和项目页。
3. 支持资料 PDF 上传到本地 `uploads/`。
4. 通过 Redis + RQ 调用 PyMuPDF 解析文字层 PDF。
5. 完成资料 chunk 化和 embedding 入库，embedding 固定使用 DashScope `text-embedding-v4`。
6. 支持用户手动输入一道题。
7. 返回最相关的资料页码、片段和原始 PDF 入口。

第一版最重要的工程原则是：

> v0.1.0 不做 OCR 和 AI 解释，只验证资料定位是否成立。

只要用户输入一道题，系统能在几秒内告诉他“看哪个文件、第几页、哪一段”，Suton 就已经具备明确价值。
