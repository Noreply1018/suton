# Suton / 溯通

Suton / 溯通 是一个面向考试复习与课程速通场景的资料溯源工具。它不试图替代学生学习，也不把自己包装成无所不知的 AI 老师，而是专注解决一个具体、高频且耗时的问题：当学生直接从做题开始复习时，如何快速找到这道题对应课本、PPT、讲义或老师资料中的哪一部分内容。

Suton 的核心价值不是制造更多泛化解释，而是降低“题目到资料”的检索成本，让刷题、查书、补知识这三个动作连成一个顺畅闭环。

## 产品定位

Suton 的核心思路是“从题目出发，反向打通资料依据”。

用户可以先上传课程资料、教材 PDF、课堂 PPT、复习讲义、实验指导书、历年题或练习题。系统通过文档解析、OCR、语义索引和大模型辅助处理，将分散资料转化为可检索的结构化知识库，并把试题拆分成可独立定位的题目对象。

当用户面对一道不会、不熟或想确认依据的题目时，Suton 不会优先给出脱离资料的泛化讲解，而是帮助用户快速定位这道题在指定资料中的相关页码、段落、公式、图表或例题。学生可以立刻回到考试依据本身，看到“这道题到底从哪里来、应该看哪几页、对应哪部分内容”。

## 解决的问题

- 资料很多：教材、PPT、讲义、实验指导书、题库和历年题分散在不同文件中。
- 考试范围明确：复习必须以课本、老师材料和课程资料为准。
- 时间有限：人工翻找题目对应知识点、页码和例题成本很高。
- 做题先行：学生常常从题目开始复习，但题目与资料依据之间缺少直接连接。
- 依据优先：相比泛泛解释，很多场景更需要知道“原文在哪里、考试依据是什么”。

## 工作流

1. 上传课程资料
   用户上传教材 PDF、课堂 PPT、复习讲义、实验指导书、历年题、练习题等资料。

2. 构建课程知识库
   系统对文档进行解析、OCR、结构化切分、语义索引和必要的大模型辅助处理。

3. 拆分题目对象
   系统将试卷、题库或练习材料拆分为可独立检索、定位和追溯的题目对象。

4. 从题目反查资料
   用户选中或输入一道题，系统在指定资料范围内检索相关页码、段落、公式、图表或例题。

5. 回到依据复习
   用户根据定位结果回到课本、PPT 或讲义原文，补齐知识点并继续刷题。

## 核心能力

- 文档解析：处理教材 PDF、课堂 PPT、讲义、实验材料等课程资料。
- OCR 识别：支持扫描版资料、图片型页面和题目截图中的文字识别。
- 语义索引：将资料内容转化为可按题目语义检索的知识库。
- 题目拆分：把试卷、练习题和历年题拆分为独立题目对象。
- 资料溯源：从题目反向定位到相关页码、段落、公式、图表或例题。
- 范围约束：优先在用户指定的课程资料和考试范围内查找依据。
- 大模型辅助：用于增强解析、匹配、归纳和检索结果组织，但不替代原始资料依据。

## 适用场景

- 考前复习：快速确认题目对应的教材章节、PPT 页码或讲义段落。
- 课程突击：在时间有限时，从高频题目反向补齐关键知识位置。
- 开卷资料整理：把分散资料与题目建立连接，减少临场翻找成本。
- 题库反查：将练习题、历年题与课程资料中的公式、概念和例题关联起来。
- 资料密集课程：适合教材、PPT、讲义、实验指导和老师补充材料较多的课程。

## 产品边界

Suton 不是一个替你学习的工具，而是一个帮你更快找到学习依据的工具。

它不会把目标放在替代学生理解、记忆和练习上，也不会默认输出脱离课程资料的“万能答案”。Suton 更关注依据、页码、段落、公式、图表和例题之间的可追溯关系，让学生能够回到原始资料中学习，并基于明确来源完成复习。

## 命名含义

Suton 有“速通”的发音意味，强调快速抵达目标知识位置。

中文名“溯通”表达了“溯源”和“打通”的双重含义：从一道题向上追溯到资料来源，再把分散的课本、讲义、PPT 与试题之间的关系打通。

## 第一版方向

第一版建议优先做 Web App，暂时不做 APK。

原因是 Suton 的第一阶段核心不是手机拍题，而是上传资料、上传或输入试题、批量解析、建立索引，并在做题时快速定位资料依据。这类工作天然依赖 PDF 预览、页码跳转、左右分栏、批量上传、后台 OCR 和索引构建，电脑端 Web 体验更顺手，工程上也更容易把核心链路打穿。

APK 可以作为后续补充形态，用于拍照上传题目、移动端碎片化刷题或考前查看资料位置，但不应成为第一版主形态。

第一版的一句话定义是：

> 上传课本、课程资料和题目后，系统自动或半自动把每道题关联到资料中的相关页码、段落和原页位置。

第一版页面只回答一个问题：

> 这道题应该去资料哪里看？

## MVP 范围

第一版应保持克制，优先验证“题目能否准确定位到资料依据”。

第一阶段只做：

- 上传资料 PDF。
- 解析 PDF 文字层，必要时对扫描页做 OCR。
- 按页保存原页截图和提取文本。
- 对资料按页、段落、标题或小节切块。
- 为资料块建立 embedding 和检索索引。
- 用户手动输入或粘贴一道题。
- 返回最相关的 3 到 5 个资料位置，包括文件名、页码、命中片段和原页入口。

第一阶段暂不做：

- APK。
- AI 讲解。
- 错题本。
- 知识图谱。
- 学习计划。
- 自动总结。
- 社区题库。
- 完整自动切题。

更极简的验证版本可以连试题上传都不做：先支持上传资料 PDF、建立索引、手动输入题目并返回资料页码、片段和截图。只要这个链路成立，产品核心价值就成立。

## 第一版页面

建议保留四个核心页面。

### 项目页

一个考试或课程对应一个项目，例如“高性能计算速通”“电磁场数值分析速通”“通信原理期末复习”。

项目下管理两类文件：

- 资料文件。
- 试题文件。

### 资料上传页

用于上传课本 PDF、课堂 PPT 导出的 PDF、老师讲义、实验指导书、扫描资料等。第一版可先只支持 PDF，后续再扩展 Word、PPT 和图片。

上传后展示处理状态：

```text
文件              页数  OCR 状态  索引状态
高性能计算课件.pdf   82  完成      完成
实验指导书.pdf       31  完成      完成
```

### 试题上传页

用于上传试题 PDF、作业题 PDF、历年真题或图片题。完整版本中，系统会自动切题并生成题目卡片。

但第一版建议先降低风险：支持用户手动粘贴题目，或从试题 PDF 中手动创建题目。自动切题容易拖慢进度，应该在资料检索准确性验证通过后再做。

### 做题与查依据页

这是主页面。建议采用左右分栏：

- 左侧展示题目。
- 右侧展示资料依据。

右侧结果不写长篇解释，只展示可追溯依据：

```text
1. 《实验指导书》第 7 页
   命中段落：关于二维块循环分布和进程网格的说明
   操作：查看原页截图

2. 《课堂 PPT》第 15 页
   命中内容：MB、NB 对通信与计算负载的影响
   操作：跳转 PDF 第 15 页
```

页面上最多保留一句提示：

> 系统判断本题主要关联这些资料位置，请优先查看。

## 技术方案

推荐第一版采用 Web + 后端处理服务 + 数据库 + 文件存储的架构。

```text
前端：Next.js / React + Tailwind CSS
后端：Python FastAPI
数据库：PostgreSQL + pgvector
后台任务：Redis + Celery 或 RQ
文件存储：本地 uploads，正式部署可迁移到 MinIO / S3 / OSS / COS
PDF 解析：PyMuPDF / Docling
OCR：PaddleOCR
LLM：仅用于切题辅助、关键词扩展、候选重排和极短命中理由
Embedding：OpenAI embedding / bge-m3 / text2vec
Reranker：bge-reranker 或兼容的重排模型
```

技术取舍原则：

- 前端优先服务上传、预览、左右分栏、题目卡片、资料依据展示和页码跳转。
- 后端负责文件接收、PDF 解析、OCR、题目处理、资料切块、embedding、检索、重排和返回依据。
- PostgreSQL 管项目、文件、页码、题目、chunk 和关联结果，pgvector 负责第一版向量检索。
- 数据量和并发压力变大后，再考虑将向量检索迁移到 Qdrant 等独立向量服务。
- OCR、解析和索引构建都应放在后台任务中处理，不阻塞用户页面。

## 文档解析与 OCR 策略

OCR 不应一把梭。第一版应区分资料类型：

- PDF 有文字层时，优先用 PyMuPDF、pdfplumber 或 Docling 直接提取文本。
- 扫描版 PDF、图片型页面、拍照讲义和截图题目，再使用 PaddleOCR。
- Tesseract 可作为低成本兜底，但不建议作为中文试题和复杂版面的主力 OCR。

第一版不要追求完美还原公式、表格和复杂排版。更重要的是保存原页截图，并让 OCR 文本辅助检索。只要系统能把题目定位到正确页码和大致段落，用户就可以回到原页自行查看细节。

## 核心处理流程

### 资料处理 Pipeline

1. 保存原始资料文件。
2. 将 PDF 按页渲染为图片，保存原页截图。
3. 判断页面是否有文字层，有则直接提取，没有则 OCR。
4. 保存每页文本、页码和 OCR 置信度。
5. 按页码、标题、段落、小节、图表说明和公式附近文字切块。
6. 为每个 chunk 生成 embedding。
7. 将 chunk、页码、原文片段、可选 bbox 和 embedding 入库。

关键数据对象：

```text
documents
- id
- project_id
- filename
- type
- status

document_pages
- id
- document_id
- page_no
- image_path
- raw_text

chunks
- id
- document_id
- page_id
- page_no
- text
- section_title
- bbox
- embedding
```

### 试题处理 Pipeline

第一版可先手动创建题目，后续再加入自动切题。

自动切题的推荐顺序是：

1. 先用规则识别题号，例如 `1.`、`一、`、`第 1 题`、`(1)`、题号加分值。
2. 规则不稳定时，再用大模型辅助识别题目边界。
3. 每道题保存题号、文本、页码和题目截图。

关键数据对象：

```text
exam_files
- id
- project_id
- filename
- status

questions
- id
- exam_file_id
- question_no
- text
- page_no
- image_path
```

## 检索与重排

只做向量检索不够。考试资料中常有强关键词和符号，例如 ScaLAPACK、MPI、MB/NB、Cholesky、FFT、抽样定理、进程网格等。第一版应采用 hybrid search：

1. 题目预处理：提取关键词、专有名词、公式符号、题型和可能章节名。
2. 关键词召回：用 PostgreSQL 全文检索或 BM25 找到精确匹配资料块。
3. 向量召回：用 embedding 找语义相近资料块。
4. 合并候选：合并关键词和向量召回结果，取前若干个 chunk。
5. 重排：用 reranker 或 LLM 判断候选资料是否确实能作为本题依据。
6. 返回依据：输出前 3 到 5 个来源明确的资料位置。

每条结果必须绑定来源：

```text
question_matches
- id
- question_id
- chunk_id
- score
- reason
- rank
```

前端展示结果时必须包含：

- 文件名。
- 页码。
- 原文片段。
- 原页截图或 PDF 跳转入口。
- 命中原因。

如果一条结果没有明确资料来源，就不应展示。

## 大模型使用边界

Suton 不是 AI 答题器，大模型不应站在台前直接讲题。

大模型在第一版只做四类辅助工作：

- 辅助切题：识别一页试卷中的题目边界。
- 提取检索关键词：把题目改写成更适合检索的 query。
- 判断资料相关性：对候选 chunk 做重排。
- 生成极短命中理由：说明为什么某页资料被推荐为依据。

所有大模型输出都必须回落到资料来源。没有 `document_id`、`page_no`、`chunk_id`、`source_text` 和 `score` 的结果，不进入前端依据列表。

## 开发顺序

建议按以下顺序推进：

1. 只支持资料 PDF 上传。
2. 完成 PDF 按页解析和原页截图。
3. 完成资料 chunk 化和 embedding 入库。
4. 支持用户手动输入一道题。
5. 返回最相关的资料页码、片段和原页截图。
6. 加入试题 PDF 上传和手动创建题目。
7. 在检索准确性稳定后，再加入自动切题。

第一版最重要的工程原则是：

> OCR 可以不完美，解释可以没有，但资料定位必须准。

只要用户输入一道题，系统能在几秒内告诉他“看第几页、哪一段、哪张图”，Suton 就已经具备明确价值。
