# Spec 撰写规则

本文档不是 Suton / 溯通的产品需求文档，也不是某个阶段的功能清单。本文档用于约束 `docs/spec/` 目录下后续 spec 文档的写法、结构、边界和验收口径。

所有 spec 都必须围绕“可实现、可测试、可追溯”的要求撰写，避免把想法、宣传语、实现细节和未验证假设混在一起。

## 基本原则

- 使用中文撰写。
- 面向实现和验收，不写成市场介绍、愿景说明或泛泛方案。
- 每份 spec 只描述一个明确主题，例如一个功能、一个页面、一个接口、一条 pipeline 或一个数据模型。
- 先写目标和边界，再写行为、数据、流程和验收。
- 所有要求必须可验证，避免使用“更好”“智能”“高效”“友好”等无法单独验收的形容词。
- 不把规划写成已实现事实。
- 不在 spec 中承诺外部系统、模型、OCR 或第三方服务一定可用，除非已有可追溯证据。

## Suton 产品边界

Suton / 溯通是资料溯源工具，不是 AI 答题器。

撰写 spec 时必须遵守以下边界：

- 核心目标是从题目定位到课程资料依据。
- 结果必须优先展示文件名、页码、原文片段、原页入口和命中原因。
- 大模型只能作为辅助组件，用于切题、关键词提取、查询改写、候选重排和极短命中理由。
- 不得要求系统无来源地回答题目。
- 不得要求系统编造页码、原文、公式解释或资料出处。
- 不得把 AI 讲解、错题本、知识图谱、学习计划、社区题库写入第一版必做范围，除非另有明确 spec 变更。

## 推荐目录结构

`docs/spec/README.md` 只保存本规则文档。

具体 spec 应放在独立文件中，例如：

```text
docs/spec/project.md
docs/spec/document-upload.md
docs/spec/document-processing.md
docs/spec/question-input.md
docs/spec/retrieval.md
docs/spec/source-viewer.md
docs/spec/data-model.md
docs/spec/api.md
```

文件命名应使用小写英文和连字符，避免使用空格、拼音缩写或含糊名称。

## Spec 模板

每份 spec 建议包含以下章节。若某章节确实不适用，可以省略，但不能省略“目标”“范围”“验收标准”。

```markdown
# 功能名称

## 目标

说明该 spec 要解决什么问题，以及为什么需要它。

## 非目标

明确本 spec 不解决什么，避免范围膨胀。

## 用户场景

描述用户在什么情况下触发该功能。

## 范围

列出必须支持、可以暂缓、明确不做的内容。

## 行为要求

描述页面、接口、任务或服务的具体行为。

## 数据要求

描述涉及的数据对象、字段、状态和约束。

## 状态与错误

描述状态流转、失败原因、重试策略和用户可见提示。

## 安全与隐私

描述上传资料、原文片段、模型调用、日志和权限相关要求。

## 验收标准

列出可以被测试或人工核验的通过条件。

## 开放问题

记录尚未确定、需要用户或外部系统确认的问题。
```

## 目标写法

目标应写成具体结果，而不是抽象愿景。

推荐：

```text
用户上传资料 PDF 后，系统应保存原文件、生成页面记录，并进入后台解析队列。
```

不推荐：

```text
打造智能高效的资料管理体验。
```

## 非目标写法

每份 spec 都应尽量写清非目标。非目标用于保护第一版范围，避免功能自然扩张。

示例：

```text
本 spec 不包含 PPT、Word 和图片上传；这些格式应在 PDF 链路稳定后单独定义。
```

```text
本 spec 不要求系统生成题目答案，只要求返回资料依据。
```

## 范围写法

范围建议分三类：

- 必须支持：本 spec 完成时必须具备。
- 可以暂缓：合理但不阻塞当前目标。
- 明确不做：容易被误加进来、需要主动排除。

示例：

```markdown
### 必须支持

- 上传 PDF 文件。
- 展示处理状态。
- 保存原始文件路径。

### 可以暂缓

- 批量上传。
- 拖拽排序。

### 明确不做

- Word 解析。
- 自动总结。
- AI 讲解。
```

## 行为要求

行为要求应描述输入、处理、输出和边界情况。

至少说明：

- 用户如何触发。
- 系统如何响应。
- 成功后产生什么结果。
- 失败时如何表现。
- 是否有后台任务。
- 是否有重试或取消。
- 用户能看到哪些状态。

不要只写“支持上传”“支持检索”这种单句要求。

## 数据要求

数据要求应明确实体、字段、关系和约束，但不要过早绑定具体 ORM 或数据库迁移语法。

推荐格式：

```text
documents
- id
- project_id
- filename
- content_type
- storage_path
- status
- created_at
- updated_at
```

字段说明需要补充时，可以在字段列表后写约束：

```text
status 只能取 waiting、processing、completed、failed。
```

## 状态与错误

涉及上传、解析、OCR、索引、检索、模型调用的 spec 必须写状态与错误。

至少考虑：

- 等待处理。
- 处理中。
- 完成。
- 失败。
- 用户取消。
- 可重试失败。
- 不可重试失败。

错误信息应可定位问题，但不暴露密钥、完整本地路径或敏感原文。

## 检索与来源约束

涉及题目检索、资料匹配、重排或结果展示的 spec 必须写来源约束。

可展示的资料依据必须至少包含：

- 文件 ID 或文件名。
- 页码。
- chunk ID 或可定位片段 ID。
- 原文片段。
- 评分或排序依据。
- 原页查看入口。

如果结果没有明确来源，不得进入前端依据列表。

## 大模型使用约束

涉及大模型的 spec 必须明确：

- 模型输入是什么。
- 模型输出用于什么。
- 输出是否需要结构化。
- 输出如何被校验。
- 失败时是否降级。
- 是否允许调用外部模型服务。
- 是否会发送用户上传资料原文。

默认规则：

- 大模型不得直接生成无来源答案。
- 大模型输出不得覆盖数据库中的原文来源。
- 大模型生成的命中原因必须绑定具体资料片段。
- 大模型失败不应导致已有资料来源被伪造或补写。

## 安全与隐私

涉及用户上传资料的 spec 必须考虑隐私。

至少说明：

- 文件保存位置。
- 文件访问权限。
- 是否记录原文日志。
- 是否向外部模型或 OCR 服务发送内容。
- 如何避免密钥、路径、用户资料泄露到日志。
- 删除项目或文件时应删除哪些派生数据。

## 验收标准

验收标准必须可测试。

推荐：

```text
上传一个含文字层的 PDF 后，系统应在页面展示文件名、页数和 completed 状态。
```

```text
输入一道题后，返回的每条依据都必须包含文件名、页码、命中片段和原页入口。
```

不推荐：

```text
检索结果足够智能。
```

```text
用户体验很好。
```

## 开放问题

无法确认的事项必须写入开放问题，不得擅自假设。

常见开放问题包括：

- 是否允许调用外部 OCR 或模型服务。
- 是否需要完全本地化部署。
- 上传文件大小限制。
- 课程资料是否需要跨项目共享。
- 是否需要多用户和权限隔离。
- 检索延迟的目标值。
- 第一版是否支持试题 PDF 上传。

开放问题应写成需要确认的具体问题，而不是笼统备注。

## 禁止事项

spec 中禁止出现以下内容：

- 将未实现能力写成已完成。
- 用宣传语替代需求。
- 无来源地要求 AI 回答题目。
- 把 APK、错题本、知识图谱、学习计划等非第一版目标混入必做范围。
- 不说明失败状态和验收标准。
- 不说明资料来源约束。
- 把第三方服务能力当作无需验证的事实。
- 把 README 中的产品定位整段复制为功能 spec。

## 与 README 的关系

根目录 `README.md` 描述产品定位、第一版方向和总体技术取舍。

`docs/spec/` 下的具体 spec 应把 README 中的方向拆成可执行、可测试、可审计的要求。若 spec 与 README 发生冲突，应先更新产品决策，再同步修改相关文档，不得让两个文档长期表达不一致。
