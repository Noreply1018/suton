# Suton Spec 总纲

本文档定义 Suton / 溯通各版本 spec 的统一写作规则、范围边界、验收要求和发布门禁。任何版本目录（例如 `docs/spec/v0.1.0`）都必须按照本文档组织内容。

## 基本原则

- 凡是记录在版本 spec 中的条目，都必须在该版本实现。
- spec 不使用 P0/P1/P2、可选优先级或可延期范围；需要延期的内容不得写入当前版本 spec，必须放入后续版本或明确列入“不包含的范围”。
- spec 不只描述“要做什么”，还必须描述“做到什么程度才算完成”。
- 每个条目都必须有明确的验收标准和验证矩阵。
- 最终验收必须基于真实 Web 应用、真实后端服务、真实数据库和可追溯的资料处理结果，不允许只依赖 mock、dry-run、静态检查或未运行服务的本地判断作为完成依据。
- 涉及 OCR、文档解析、embedding、重排、大模型、对象存储、队列、数据库、CI/CD 或远端仓库的结论，必须通过可追溯证据核验。
- 版本范围必须可追溯变更，不能通过口头约定新增、删除或降级条目。

## Suton 产品边界

Suton 是资料溯源工具，不是 AI 答题器。

所有版本 spec 都必须遵守以下边界：

- 核心目标是从题目定位到课程资料依据。
- 可展示结果必须绑定文件名、页码、原文片段、原页入口和命中原因。
- 大模型只能用于切题辅助、关键词提取、查询改写、候选重排和极短命中理由。
- 不允许无来源地回答题目。
- 不允许编造资料页码、原文片段、公式解释或资料出处。
- 不允许把 AI 讲解、错题本、知识图谱、学习计划、社区题库等非当前版本目标写入当前版本必做范围。

## 版本 Spec 必备结构

每个版本目录至少包含以下内容：

```text
docs/spec/<version>/
├── README.md
└── items/
```

其中 `README.md` 是版本总览，`items/` 存放具体条目。版本总览至少包含：

1. 版本目标
2. 范围分类
3. 行为兼容与约束
4. 明确不包含的范围
5. 验证总规则
6. 发布门禁
7. Spec 变更记录

## 版本目标

版本目标必须说明：

- 本版本要解决的核心问题。
- 本版本面向的用户或使用场景。
- 本版本必须交付的用户可见结果。
- 本版本明确不解决的问题。
- 后续版本才会处理的能力。

版本目标用于约束版本边界，避免 spec 自然膨胀。

## 范围分类

版本 spec 的条目必须至少归入以下分类之一：

- 问题修复
- 设计优化
- 功能新增
- 行为兼容与约束
- 数据与接口变更
- 处理链路变更
- 验证与发布门禁

版本总览中还必须有“明确不包含的范围”章节，用于记录本版本不做的事项。凡是未进入范围且未通过变更规则加入的内容，默认不属于当前版本。

## 条目模板

每个条目必须使用统一模板：

```markdown
# 条目名称

- 类型：问题修复 / 设计优化 / 功能新增 / 行为兼容与约束 / 数据与接口变更 / 处理链路变更 / 验证与发布门禁
- 状态：草案 / 已定稿 / 实现中 / 已完成 / 已移除
- 背景：
- 当前问题：
- 目标行为：
- 非目标：
- 发布必要性：必须发布
- 用户可见影响：
- 涉及模块：
- 配置、接口或数据结构变化：
- 兼容性要求：
- 验收标准：
- 验证矩阵：
- 风险与回滚：
```

关键字段要求：

- `目标行为` 必须描述实现后的可观察行为。
- `非目标` 必须说明本条目不会处理的边界。
- `发布必要性` 必须写“必须发布”；如果某事项不是当前版本发布前必须完成的门禁项，不得写入当前版本条目。
- `用户可见影响` 必须说明用户在页面、结果、状态、错误提示或资料依据中能观察到什么变化；如无用户可见影响，必须明确写“无用户可见影响”。
- `涉及模块` 必须明确到前端页面、后端 API、后台任务、数据库、检索、OCR、文件存储、部署或文档。
- `验收标准` 必须能被人工判断。
- `验证矩阵` 必须能被机器、真实服务或真实环境复现。
- `配置、接口或数据结构变化` 如无变化，必须明确写“无”。
- `兼容性要求` 如无特殊要求，必须明确写“无特殊要求”。

## 验证矩阵

每个条目必须包含验证矩阵，格式如下：

```markdown
| 场景 | 环境 | 前置条件 | 操作命令 | 预期结果 | 实际结果 | 证据 | 结论 |
|---|---|---|---|---|---|---|---|
```

字段要求：

- `场景`：每个条目至少覆盖正常路径、异常路径、回归或兼容路径。涉及外部依赖、数据迁移、权限、配置、接口、数据结构、并发、性能、OCR、模型调用、文件存储或后台任务风险时，必须增加对应场景。
- `环境`：必须说明 Node.js、包管理器、Python、数据库、Redis、对象存储、OCR/模型服务、浏览器、操作系统、关键环境变量和配置。
- `前置条件`：说明测试数据、上传文件、数据库状态、账号、密钥、外部依赖状态和远端服务状态。
- `操作命令`：必须给出可复现命令；如果需要人工操作，必须写清步骤。
- `预期结果`：必须具体到可观察输出、界面行为、API 响应、任务状态、数据库变化、检索结果、日志、文件产物或资料来源。
- `实际结果`：实现验证时填写真实结果。
- `证据`：记录命令输出、测试报告、截图、录屏、日志路径、数据库查询结果、API 响应、CI 链接、远端查询结果或其他可追溯材料。
- `结论`：只能填写通过、失败、阻塞或移除。

## 真实 Web 与处理链路验证标准

最终验收必须满足以下要求：

- 使用项目正式构建、启动和测试命令，不允许使用专门绕过依赖、绕过状态、绕过权限或替代真实服务的测试入口作为唯一验证依据。
- 前端功能必须在真实浏览器或可证明等价的浏览器自动化环境中验证。
- 后端 API 必须通过真实运行的后端服务验证，不允许只依赖函数级 mock 证明完成。
- 数据库相关功能必须连接真实数据库实例验证，并记录迁移、查询或数据变化证据。
- 后台任务相关功能必须通过真实队列或可证明等价的本地队列执行路径验证，并记录任务入队、执行、完成或失败状态。
- 文件上传、资料解析、OCR、页图渲染、chunk 化、embedding、检索和重排链路，必须使用真实样例文件或版本 spec 指定的固定测试资料验证。
- 资料依据展示必须能追溯到具体文件、页码、chunk 或原文片段；没有来源证据的结果不得标记为通过。
- 涉及大模型、OCR、embedding、reranker 或外部对象存储时，必须记录调用方式、输入范围、输出摘要、失败处理和是否发送用户资料原文。
- 验证前必须清理会影响结论的旧状态，避免依赖历史上传、缓存、旧数据库记录、旧索引或旧任务结果。
- 验证记录必须包含输入、执行命令、预期结果、实际结果和证据。

如果某个条目无法在本机完成真实 Web 构建、服务启动、数据库验证、文件处理、模型调用或外部系统核验，或关键事实、关键步骤、外部系统状态无法可靠确认，必须标记为阻塞，并说明：

- 已尝试的核验路径。
- 合理替代路径。
- 失败证据。
- 缺失条件。
- 需要用户补充的权限、凭据、网络、设备、环境、工具、测试资料或人工证据。

外部系统状态包括但不限于远端仓库、Git tag、GitHub Release、CI/CD 运行结果、部署环境、对象存储、OCR 服务、模型服务、向量服务、数据库实例、Redis、第三方 API、权限凭据和网络可达性。涉及这些状态时，必须通过可追溯证据核验，例如 `git ls-remote`、`gh release view`、CI 运行记录、平台 API 查询、数据库查询、对象存储列表、服务健康检查、用户提供的可信截图或日志。

不得将未完成或未核验的条目标记为完成。

## 资料来源与模型输出门禁

涉及题目检索、资料匹配、结果展示或大模型输出的条目，必须额外满足：

- 每条展示给用户的资料依据必须包含文件名或文件 ID、页码、chunk ID 或可定位片段 ID、原文片段、评分或排序依据、原页入口。
- 命中原因必须基于具体资料片段生成，不得脱离资料来源泛化解释。
- 大模型输出不得覆盖数据库中的原文来源。
- 大模型失败时不得伪造依据、补写页码或把无来源内容展示给用户。
- 如果返回结果为空，必须清楚展示空状态，不得用 AI 猜测补齐。

## 发布门禁

版本发布前必须满足：

- 所有已定稿条目均已实现。
- 所有验证矩阵均已执行并通过。
- 所有失败项均已修复，或通过 spec 变更规则正式移除。
- 文档与实际行为一致。
- 真实 Web 构建、服务启动、数据库连接、必要 API、必要页面和关键资料处理链路验证通过。
- 关键外部系统状态已有可追溯证据。
- 资料来源与模型输出门禁已通过。
- Git 工作区中与发布相关的改动已完成 subagent 严格审计，且审计结论为通过。
- 如 subagent 审计不通过，必须反复修改并重新审计，直到通过后才允许提交。
- 在 Git 仓库中，每轮对已追踪文件完成改动并通过审计后，必须提交相关改动。

## Spec 变更规则

版本 spec 定稿后，变更必须遵守：

- 新增条目默认进入后续版本。
- 如必须加入当前版本，必须同时补充验收标准和验证矩阵。
- 删除或降级条目必须记录原因、影响和批准依据。
- 批准依据必须写入版本总览的变更记录，至少包含批准人或来源、批准时间、批准方式，以及可追溯证据链接、命令输出、截图、日志路径或用户确认原文。
- 所有范围变更必须写入版本总览的变更记录。
- 不允许通过口头约定改变当前版本范围。

## 文件命名

条目文件必须放在版本目录的 `items/` 下，并使用稳定编号：

```text
items/
├── fix-001-example.md
├── design-001-example.md
├── feature-001-example.md
├── compat-001-example.md
├── data-001-example.md
├── pipeline-001-example.md
└── gate-001-example.md
```

编号只用于稳定引用，不代表优先级、处理顺序或可延期程度。
