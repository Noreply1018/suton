# Suton / 溯通 Spec 要求

本文档定义 Suton / 溯通第一版的产品、功能、数据和验收要求。它面向后续设计、开发和测试，不描述已经完成的实现。

## 目标

Suton 是一个面向考试复习与课程速通场景的资料溯源工具。第一版只解决一个核心问题：

> 用户输入或选中一道题后，系统告诉用户应该去课程资料的哪一页、哪一段、哪张图或哪个例题查看依据。

第一版不以“讲题”“代学”“生成答案”为目标。系统可以使用大模型辅助解析和检索，但所有可展示结果必须绑定用户上传资料中的明确来源。

## 第一版形态

第一版采用 Web App，不做 APK。

Web App 是第一版主形态，因为本产品的核心操作包括资料上传、PDF 预览、页码跳转、左右分栏查看、后台 OCR、索引构建和批量处理。APK 可以作为后续补充，用于拍照上传题目、移动端查看资料位置和碎片化刷题，但不进入第一版范围。

## 用户与场景

目标用户是需要依据课程资料复习的学生，尤其是：

- 资料多：课本、PPT、讲义、实验指导书、复习资料、历年题分散存在。
- 范围明确：考试以老师资料、课本和课程文件为准。
- 时间有限：考前需要从题目快速回到资料依据。
- 做题先行：先刷题，再按题目反查知识来源。

典型场景：

- 考前复习时，输入一道不会的题，快速找到对应资料页码。
- 开卷考试整理资料时，把题目与资料位置建立关联。
- 课程突击时，从高频题反向补齐课本或 PPT 中的关键内容。
- 题库反查时，确认某道题主要来自哪份讲义、哪页 PPT 或哪段教材。

## 范围

### 第一阶段必须支持

- 创建课程或考试项目。
- 上传资料 PDF。
- 保存资料原文件。
- 将 PDF 按页渲染为原页图片。
- 提取 PDF 文字层。
- 对无文字层页面或扫描页执行 OCR。
- 保存页码、原页图片、页面文本和 OCR 状态。
- 将资料按页、标题、段落、小节或相邻语义块切分为 chunk。
- 为 chunk 建立 embedding 和检索索引。
- 用户手动输入或粘贴一道题。
- 检索并返回 3 到 5 个最相关资料位置。
- 每个结果展示文件名、页码、命中片段、命中原因和原页入口。

### 第一阶段可以暂缓

- 试题 PDF 上传。
- 用户从试题 PDF 中手动创建题目。
- 自动切题。
- 题目截图裁剪。
- bbox 高亮。
- PPT、Word、图片直接上传。

### 第一阶段明确不做

- APK。
- AI 长篇讲解。
- AI 直接给答案。
- 错题本。
- 知识图谱。
- 学习计划。
- 自动总结。
- 社区题库。
- 多人协作。
- 复杂权限系统。

## 页面要求

### 项目页

项目表示一个课程或一次考试复习任务。

项目页应展示：

- 项目名称。
- 资料文件数量。
- 题目数量。
- 最近处理状态。
- 进入资料上传、题目输入和查依据页面的入口。

### 资料上传页

资料上传页用于上传课程资料 PDF。

上传后应展示：

- 文件名。
- 页数。
- 解析状态。
- OCR 状态。
- 索引状态。
- 失败原因。

状态至少包括：

- 等待处理。
- 处理中。
- 完成。
- 失败。

### 题目输入页

第一版以手动输入或粘贴题目为主。

页面应支持：

- 输入题目文本。
- 选择检索范围，默认在当前项目全部资料内检索。
- 提交检索。
- 查看历史题目。

### 查依据页

查依据页是核心页面，应采用左右分栏。

左侧展示：

- 题目文本。
- 题目编号或用户自定义标题。
- 所属项目。

右侧展示资料依据列表：

- 排名。
- 文件名。
- 页码。
- 命中片段。
- 命中原因。
- 相似度或综合分数。
- 查看原页按钮。
- 跳转 PDF 页码按钮。

右侧不展示长篇 AI 讲解。可以展示一句极短提示：

> 系统判断本题主要关联这些资料位置，请优先查看。

- `document_id`
- `page_no`
- `source_text`
- `score`

缺少来源约束的结果不得展示。

## 处理流程

### 资料处理

1. 用户上传资料 PDF。
2. 系统保存原文件。
3. 系统创建后台任务。
4. 后台任务读取 PDF 页数。
5. 后台任务按页渲染原页图片。
6. 后台任务判断每页是否有文字层。
7. 有文字层时直接提取文本。
8. 无文字层或文字过少时执行 OCR。
9. 系统保存页面文本和 OCR 状态。
10. 系统按页、标题、段落、小节或相邻语义块切分 chunk。
11. 系统生成 embedding。
12. 系统写入检索索引。
13. 系统将文件状态更新为完成或失败。

### 题目检索

1. 用户输入题目。
2. 系统保存题目记录。
3. 系统提取关键词、专有名词、公式符号和可能章节名。
4. 系统执行关键词召回。
5. 系统执行向量召回。
6. 系统合并候选 chunk。
7. 系统执行重排。
8. 系统生成极短命中原因。
9. 系统保存 `question_matches`。
10. 前端展示来源明确的前 3 到 5 条结果。

## 检索要求

第一版应使用 hybrid search，不允许只依赖向量检索。

检索至少包括：

- 关键词召回：用于保留精确术语、英文缩写、公式符号和课程特定名词。
- 向量召回：用于找到语义相近但字面不完全一致的资料块。
- 候选合并：合并关键词和向量结果，去重后保留候选集。
- 重排：判断候选资料是否确实能作为题目依据。

返回结果应优先满足：

- 来源明确。
- 页码准确。
- 片段相关。
- 命中原因简短可读。
- 可打开原页核验。

## 大模型边界

大模型只能用于辅助，不得替代资料来源。

允许使用：

- 题目关键词提取。
- 查询改写。
- 自动切题辅助。
- 候选资料重排。
- 极短命中原因生成。

禁止使用：

- 无来源直接回答题目。
- 编造资料页码。
- 编造原文片段。
- 输出脱离用户资料的长篇讲解。
- 在没有 `document_id`、`page_no`、`chunk_id` 和 `source_text` 的情况下生成前端依据结果。

## 技术约束

推荐技术栈：

```text
Frontend: Next.js / React + Tailwind CSS
Backend: Python FastAPI
DB: PostgreSQL + pgvector
Worker: Redis + Celery 或 RQ
Storage: 本地 uploads，后续可迁移到 MinIO / S3 / OSS / COS
PDF Parser: PyMuPDF / Docling
OCR: PaddleOCR
Embedding: OpenAI embedding / bge-m3 / text2vec
Reranker: bge-reranker 或兼容重排模型
```

第一版本地开发可以使用本地文件系统存储上传文件。正式部署时，文件存储应抽象为可替换接口，便于迁移到对象存储。

OCR 和索引构建必须是后台任务，不应阻塞页面请求。

## 验收标准

第一版 MVP 通过的最低标准：

- 用户可以创建项目。
- 用户可以上传资料 PDF。
- 系统可以展示资料处理状态。
- 系统可以提取至少一份 PDF 的页面文本。
- 系统可以保存每页原页图片。
- 系统可以对资料建立 chunk 和检索索引。
- 用户可以手动输入一道题。
- 系统可以返回 3 到 5 条资料依据。
- 每条依据都有文件名、页码、片段、命中原因和原页入口。
- 没有明确来源的结果不会展示。
- 页面不输出长篇 AI 答案。

核心验收原则：

> OCR 可以不完美，解释可以没有，但资料定位必须准。

如果系统能在用户输入题目后，于可接受时间内返回可核验的资料页码、片段和原页入口，即视为第一版核心链路成立。
